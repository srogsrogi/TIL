# API 설계 및 문서 표준

- 웹 API 설계 원칙(James Higginbotham) 책 읽음
- 원래 알던 내용들은 제외하고 새로 알게 된 내용이나 내 생각 위주로 간단하게 정리
- 특히 문서화 표준 공부하고 팀이랑 공유해서 잘 써볼 예정



## 설계 원칙

- 한 번 정하면 바꾸기 매우 어렵다. 마감에 쫓겨 급하게 작성하면 반드시 부채가 됨
-  REST가 가장 좋고 유일한 표준은 아니다. 



## 문서 표준

### OAS

- OpenAPI Specification

- RESTful API를 위한 표준 명세로, 가장 널리 사용되는 산업 표준
- JSON 또는 YAML 사용
- 코드 생성(Code Generation) 도구가 강력함

### RAML

- RESTful API Modeling Language

- API의 라이프사이클 전체를 모델링하기 위해 설계된 언어
- YAML 사용

- 계층적 구조로 사람이 쓰고 읽기 좋음
- 재사용성(Traits, Resource Types)이 뛰어남
- 설계 중심적인 접근 방식



## 비교 및 선택 기준

| 특징 | OAS (OpenAPI) | RAML |
| :--- | :--- | :--- |
| **주요 목적** | 문서화 및 상호운용성 | 설계 및 모델링 |
| **인기도** | 매우 높음 (사실상 표준) | 줄어드는 중 |
| **도구 지원** | 매우 풍부함 | 있긴 함         |
| **문법** | 다소 장황할 수 있음 | 간결하고 계층적 |
